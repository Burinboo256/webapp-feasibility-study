Diagnosis Feasibility Study Webapp

Project Overview

This project is a Node.js web application for cohort feasibility research. It supports diagnosis, laboratory, and medication criteria, generates SQL Server style feasibility SQL, and can run in either local demo mode or SQL Server backed mode.

The current builder uses a reusable Airtable style advanced filter UI for T0 index event conditions, inclusion criteria, and exclusion criteria. Users can create nested AND or OR groups, add or remove condition rows, and build reusable field or operator based logic without inline concept preview widgets.

Repository

Primary Commands

- pnpm install
- pnpm dev
- pnpm test
- pnpm sync-dictionary
- node scripts/hash-password.mjs "new-password"

What The App Does

- Builds cohort definitions with T0 index events, demographics, inclusion rules, exclusion rules, nested condition groups, timing windows, and typed field operators.
- Runs feasibility counts through the Node backend instead of directly in the browser.
- Generates SQL Server style CTE-based SQL for review and debugging.
- Supports login, OTP flows, saved cohorts, and audit logging.
- Provides a separate master dictionary page for diagnosis, lab, and drug code review before adding conditions.

First-Time Local Setup

1. Install Node.js 20 or newer.
2. Install dependencies with pnpm install.
3. Copy the example files to local runtime files.
4. If needed, copy public/data/synthetic-clinical-data_example.json to public/data/synthetic-clinical-data.json.
5. Start the app with pnpm dev.

The application can boot from the example config and example users data when local files are absent, but real local work should still use the copied local files above.

Main Config Template

The checked-in config template is:

config/app.config.example.json

Create your local config/app.config.json from that file before editing any environment-specific values.

This configuration controls:

- server host, port, and cookie security
- OTP and session settings
- Google OAuth settings
- SMTP settings
- clinical data source selection
- app storage mode
- SQL Server connection settings

Important Config Switches

The application uses two independent switches:

- clinicalDataSource: "json" or "sqlserver"
- appStorage: "local" or "sqlserver"

Recommended combinations:

- Demo or local development:
  clinicalDataSource = "json"
  appStorage = "local"

- Production-like deployment:
  clinicalDataSource = "sqlserver"
  appStorage = "sqlserver"

Meaning Of Each Switch

- clinicalDataSource controls where feasibility counts and cohort execution data come from.
- appStorage controls where users, sessions, pending OTPs, saved cohorts, feasibility run logs, and audit session logs are stored.

Authentication And Audit

- The app requires login before using the cohort builder, dictionary page, and logs page.
- Credentials login and Google OAuth are both supported.
- OTP flows are used for sign-up confirmation and password reset.
- If SMTP host is blank, OTPs are written to the server console for development.
- If SMTP delivery fails in local mode, the app falls back to the server console with a warning.
- Feasibility run logs include user identity, question, counts, attrition, full cohort config, and generated SQL.
- Audit session logs track session activity, page views, and run counts.

Current Scope

- Reusable T0, inclusion, and exclusion rule editors
- AND and OR logic in every group

Master Dictionary

The master dictionary page is used for search and review before adding conditions.

- Search by code, name, group, or count
- Filter by diagnosis, lab, or drug domain
- Copy code or name from a result card
- Read from the local snapshot file public/data/master-dictionary.json

The local snapshot is generated from these Google Sheet sources:

- ICD-10
- ICD-9
- Lab
- Drug

Refresh the snapshot with:

pnpm sync-dictionary

Saved Cohorts

- Users can save the current cohort definition, list saved cohorts, reload them, and delete them.
- Saved cohorts are handled through backend APIs.
- In local mode they are stored in data/saved-cohorts.json on the server.
- In SQL mode they are stored in SQL Server application tables.

Generated SQL

The SQL builder produces SQL Server style feasibility SQL with these assumptions:

- base table: Patient_Info
- patient key: OH_PID
- diagnosis table: Diagnosis
- laboratory table: Laboratory
- medication table: Medication
- style: CTEs plus EXISTS and NOT EXISTS
- date functions: DATEDIFF, DATEADD, and BETWEEN

The generated SQL is a feasibility draft and should be reviewed against the real source schema before production use.

Cloud And Production Notes

- This app must run with the Node server in scripts/dev-server.mjs.
- Do not deploy it as static files only if login, OTP, audit, or saved cohorts are required.
- For real deployment, create local config from the example, then set SMTP, Google OAuth, and SQL Server values outside Git-tracked files.
- Replace demo credentials before real use.
- Shared SQL-backed storage is the correct direction for multi-user environments.
